# SynapseLab 答辩展示说明

## 一、答辩时你要讲清楚什么

答辩老师最关心的不是你会不会背代码，而是：

- 你为什么做这个题目
- 这个系统解决了什么问题
- 你做了哪些功能
- 技术方案为什么这样选
- 你到底做了多少工作量
- 系统是否真的能运行

所以答辩时不要从代码文件讲起，而是从“问题、方案、实现、结果”四个层次讲。

## 二、推荐展示顺序

### 1. 课题背景

推荐讲法：

随着复杂科研问题越来越具有跨学科特征，传统依赖邮件、微信群和分散文档的协作方式效率较低。不同学科研究者之间存在术语壁垒、资源壁垒和沟通壁垒，因此有必要设计一个支持科研猜想交流、项目协作推进和人员连接的跨学科科研协作平台。

### 2. 课题目标

推荐讲法：

本项目希望构建一个基于 React 和 CloudBase 的跨学科科研协作平台 SynapseLab，实现科研猜想发布、项目协作管理、学者网络连接、评论讨论与通知提醒等功能，并完成系统从本地开发到云端部署的全过程。

### 3. 系统架构

推荐讲法：

系统采用三层架构设计。前端采用 React 构建界面和交互，后端采用 CloudBase 云函数实现业务逻辑，数据层采用 CloudBase 云数据库存储用户、猜想、项目、评论和通知等数据。通过统一服务层封装，系统可以在 mock 模式与 cloudbase 模式之间切换，方便分阶段开发与部署。

### 4. 核心功能演示

你答辩时建议按下面顺序演示：

1. 首页
2. 思想熔炉
3. 发布新猜想
4. 猜想详情与评论
5. 研究工作室
6. 项目详情与讨论
7. 学者网络
8. 通知中心
9. 个人中心

### 5. 技术实现亮点

建议重点讲这 5 个：

1. 前后端分层清晰，前端通过统一 API 层访问后端
2. 系统支持从 mock 数据平滑切换到 CloudBase 真实环境
3. 云函数按业务模块拆分，便于维护和扩展
4. 页面具备加载态、错误态和空状态，提升可用性
5. 项目最终完成了真实云端部署，而非停留在本地原型

### 6. 测试与部署结果

推荐讲法：

本项目完成了本地运行测试、页面功能测试、云函数联调测试和 CloudBase 线上部署测试。测试结果表明，系统可以稳定完成主要页面访问和核心业务操作，满足毕业设计的软件演示要求。

## 四、老师可能会问的问题

### 1. 为什么选 React 和 CloudBase

可回答：

React 适合构建组件化前端界面，开发效率高，状态管理和页面拆分清晰；CloudBase 提供云函数、数据库和静态网站托管能力，适合中小型系统快速完成全栈开发和部署。

### 2. 为什么要先 mock 再接云端

可回答：

因为直接在云环境中同时处理前端逻辑、后端逻辑和平台配置，排错成本较高。先在本地用 mock 数据完成页面和交互，再切换到真实后端，可以显著降低开发和联调风险。

### 3. 你实现的工作量体现在哪里

可回答：

工作量主要体现在需求分析、页面设计、前端实现、服务层封装、云函数设计、数据库建模、CloudBase 部署、测试联调以及论文文档整理等多个方面，不仅完成了页面原型，还完成了真实云端部署和系统联调。

### 4. 你的系统有哪些不足

可回答：

目前系统中部分互动功能仍偏演示型，例如积分下注、成员邀请和通知联动还可以进一步做成完整持久化业务；此外，平台的权限体系、推荐算法和 AI 能力仍有进一步优化空间。

## 五、答辩时的操作建议

- 尽量使用线上地址演示，而不是本地地址
- 答辩前准备 6 到 10 张关键页面截图，防止网络抖动
- 浏览器提前登录 CloudBase 控制台，必要时展示部署结果
- 演示时只走最稳定的路径，不要临场随意乱点
- 提前准备一份“系统已实现功能清单”

## 六、你最后可以这样收尾

本课题围绕跨学科科研协作这一实际需求，设计并实现了基于 React 与 CloudBase 的 SynapseLab 平台。系统已完成前端界面、后端云函数、数据库建模、核心业务实现与云端部署。通过本次毕业设计，我完成了从需求分析、系统设计、功能实现到测试部署的完整软件开发过程，也认识到正式生产系统在安全、协作流程和智能推荐方面仍有进一步优化空间。
