# 权限与安全设计

## 权限方向

- 游客：只能浏览公开内容
- 注册用户：可发布、评论、参与项目
- 项目发起人：可管理成员和阶段
- 管理员：维护平台秩序

## 安全方向

- 用户数据最小化存储
- 敏感操作要做权限判断
- AI 输出需要风险提示

## 当前测试期实际策略

为了快速打通前后端联调，当前云函数调用权限采用了较宽松的测试规则：

```json
{
  "*": {
    "invoke": true
  }
}
```

这套规则只适合开发、演示和论文截图阶段，不适合长期作为生产环境配置。

## 正式环境建议

正式环境应将“公开浏览能力”和“写操作能力”拆开：

- 公开读取类函数允许匿名访问
- 创建、评论、邀请等写操作要求已登录用户
- 管理类函数仅允许管理员调用

推荐的函数级权限示例如下：

```json
{
  "ideas": {
    "invoke": true
  },
  "projects": {
    "invoke": true
  },
  "users": {
    "invoke": "auth != null"
  },
  "comments": {
    "invoke": "auth != null"
  },
  "project-discussions": {
    "invoke": "auth != null"
  },
  "notifications": {
    "invoke": "auth != null"
  }
}
```

上面的设计含义是：

- `ideas`、`projects` 先允许公开访问，保证平台内容可浏览
- `users`、`comments`、`project-discussions`、`notifications` 需要登录后调用
- 后续如果接入正式登录体系，再进一步细化为“仅本人”“仅项目成员”“仅管理员”

## 数据库安全建议

虽然当前项目主要通过云函数访问数据库，但仍建议在 CloudBase 中补数据库规则，避免误操作或未来直接开放客户端访问时产生风险。

建议原则如下：

1. `users`

- 公开资料字段可读
- 个人敏感字段仅本人可读写
- 积分、信誉等统计字段不允许客户端直接写入

2. `ideas`

- 公开猜想允许读取
- 创建者可编辑自己发布的草稿或未归档内容
- 热度、AI 分析结果、下注统计应由服务端维护

3. `projects`

- 公开项目允许读取
- 私有项目仅成员可读
- 项目进度、成员列表、邀请状态由项目负责人或服务端修改

4. `comments` 与 `project_discussions`

- 已登录用户可新增
- 创建者仅可修改本人内容
- 删除操作最好通过管理员审核或软删除实现

5. `notifications`

- 只允许接收者读取自己的通知
- 不允许客户端直接批量生成通知

## 安全落地清单

- 在控制台中将演示期开放规则切换为正式规则
- 为数据库补充基于用户身份和文档归属的安全规则
- 将写操作统一收口到云函数
- 对邀请成员、更新进度、下注积分等操作增加服务端校验
- 对 AI 生成内容增加“仅供参考”提示
- 在论文中说明测试期与正式期权限策略的差异
